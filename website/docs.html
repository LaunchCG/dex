dex Documentation

dex is a universal package manager for AI coding agents. It installs skills, commands, rules, and MCP servers into Claude Code, Cursor, and GitHub Copilot projects from a single HCL config.

Installation

macOS & Linux

curl -fsSL https://dexartifactsproduction.z13.web.core.windows.net/install.sh | bash

Windows (PowerShell)

irm https://dexartifactsproduction.z13.web.core.windows.net/install.ps1 | iex

Installs to ~/.bin/dex. Add ~/.bin to your PATH if needed.

Global options

FlagDescription
--help, -hShow help message
--verbose, -vIncrease verbosity (-v info, -vv debug, -vvv trace)

Quick Start

1. Initialize a project:

dex init                        # Claude Code (default)
dex init --platform cursor         # Cursor
dex init --platform github-copilot # GitHub Copilot

2. Edit dex.hcl to add plugins:

project {
  default_platform = "claude-code"
}

plugin "my-tools" {
  source  = "git+https://github.com/example/my-tools.git"
  version = "^1.0.0"
}

3. Sync:

dex sync

4. Use in your AI assistant. Skills, commands, and rules are now installed in your project.

Commands

dex init

Initialize a new dex project in the current directory. Creates a dex.hcl file.

dex init [OPTIONS]
OptionShortDescriptionDefault
--platform-PTarget AI agent platformclaude-code
--name-nProject nameDirectory name
--path-pProject directoryCurrent directory

dex sync

Synchronize plugins to match dex.hcl. Without arguments, installs missing, updates outdated, and prunes orphaned plugins.

dex sync [PLUGINS...] [OPTIONS]
OptionShortDescriptionDefault
--source-sDirect source path (file://)
--registry-rRegistry to use
--no-saveDon't save to config filefalse
--no-lockDon't update lock filefalse
--force-fOverwrite files not managed by dexfalse
--path-pProject directoryCurrent directory
--namespaceNamespace resources with package namefalse
--dry-run-nShow what would change without making changesfalse
--git-excludeUpdate .git/info/exclude to hide dex-managed filesfalse

Version specifiers

FormatExampleDescription
Exactplugin@1.2.0Exact version
Caretplugin@^1.0.0Compatible with 1.x.x
Tildeplugin@~1.2.0Compatible with 1.2.x
Rangeplugin@>=1.0.01.0.0 or higher
LatestpluginLatest version

ℹ By default, dex refuses to overwrite files that exist but aren't tracked in the manifest. Use --force to override.

# Sync all plugins
dex sync

# Preview changes without applying
dex sync --dry-run

# Install a specific plugin with version
dex sync my-plugin@^1.0.0

# Install from local source
dex sync --source file:./local-plugin

# Hide managed files from git status
dex sync --git-exclude

dex list

List installed plugins.

dex list [OPTIONS]
OptionShortDescription
--tree-tShow dependency tree
--path-pProject directory

dex uninstall

Uninstall plugins from the project.

dex uninstall PLUGINS... [OPTIONS]
OptionShortDescription
--remove-rAlso remove from dex.hcl (drop the dependency)
--path-pProject directory

Without --remove: deletes installed files but keeps the plugin in dex.hcl. With --remove: also drops the plugin from dex.hcl.

dex info

Show information about an installed plugin: version, source, lock file entry, and dependencies.

dex info PLUGIN [OPTIONS]
OptionShortDescription
--path-pProject directory

dex manifest

Show files managed by dex. Optionally filter by plugin name.

dex manifest [PLUGIN] [OPTIONS]
OptionShortDescription
--path-pProject directory

dex version

Show the dex version.

dex version
$ dex version
dex 0.1.0

Project Configuration

The dex.hcl file configures a dex-managed project.

project block

AttributeTypeRequiredDescription
namestringProject name (defaults to directory name)
default_platformstringrequiredTarget AI platform: claude-code, cursor, or github-copilot

registry block

Registries define where to fetch plugins from.

AttributeTypeRequiredDescription
namestringrequiredRegistry identifier (block label)
pathstringconditionalLocal filesystem path
urlstringconditionalRemote URL (https://, s3://, az://)

plugin block

AttributeTypeRequiredDescription
namestringrequiredPlugin identifier (block label)
sourcestringconditionalDirect source URL (git+https://, file://)
registrystringconditionalRegistry name to fetch from
versionstringVersion constraint
configmap(string)Plugin-specific configuration values
 
project {
  default_platform = "claude-code"
}

registry "community" {
  url = "https://plugins.example.com"
}

plugin "python-tools" {
  registry = "community"
  version  = "^1.0.0"
  config = { python_version = "3.12" }
}

plugin "custom-plugin" {
  source  = "git+https://github.com/user/custom-plugin.git"
  version = "v2.0.0"
}

Version Constraints

SpecifierMeaning
1.0.0Exact version
^1.0.0Compatible with 1.x.x (≥1.0.0 <2.0.0)
~1.0.0Patch-level changes (~1.0.x)
≥1.0.0Greater than or equal
latestLatest available version

Lock File (dex.lock)

Records exact versions and checksums for deterministic installs. Commit it for team projects and production deployments.

{
  "version": "1.0",
  "platform": "claude-code",
  "plugins": {
    "python-tools": {
      "version": "1.2.0",
      "resolved": "https://registry.example.com/python-tools-1.2.0.tar.gz",
      "integrity": "sha512-abc123..."
    }
  }
}

HCL Functions

file(path)

Reads the contents of a file relative to the package directory.

content = file("skills/my-skill.md")

templatefile(path, vars)

Renders a template file with Go text/template syntax.

content = templatefile("skills/my-skill.md.tmpl", {
  project_name = "MyProject"
  api_version  = "v2"
})

env(name [, default])

Reads an environment variable with an optional default value.

env = {
  API_KEY = env("MY_API_KEY")
  DEBUG   = env("DEBUG", "false")
}

Template Variables

Available when rendering templates via templatefile():

VariableDescription
{{ .ComponentDir }}Absolute path to the installed component directory
{{ .PluginName }}Name of the plugin being installed
{{ .PluginVersion }}Version of the plugin
{{ .ProjectRoot }}Absolute path to the project root
{{ .Platform }}Target platform (e.g., claude-code)

Claude Code

Personal Config

Personal config lets you add resources locally without touching the shared project config. Your additions are invisible to teammates and never land in git.

local.hcl

Place a local.hcl at ~/.dex/local.hcl to define global personal resources. Supports the same resource types as dex.hcl without the required project block.

# ~/.dex/local.hcl - applied to every dex project on this machine

plugin "my-personal-tools" {
  source  = "git+https://github.com/me/my-tools.git"
  version = "latest"
}

claude_settings "personal" {
  allow = ["Bash(gh:*)"]
}

Per-project config

For project-specific personal config, create ~/.dex/projects/<project-name>/project.hcl. The project name matches the name field in dex.hcl.

# ~/.dex/projects/my-webapp/project.hcl

claude_settings "local-dev" {
  env = {
    DATABASE_URL = "postgres://localhost/myapp_dev"
  }
}

Git exclude

Hide dex-managed files from git status without modifying .gitignore:

dex sync --git-exclude

Writes patterns to .git/info/exclude, a local gitignore that only affects your checkout. Files remain on disk and fully functional.

ℹ --git-exclude is local only. Teammates see the files normally unless they also use this flag.

Variable Precedence

When multiple sources define the same variable, dex resolves them in this order (highest wins):

  1. Project variables in dex.hcl
  2. Per-project local variables in ~/.dex/projects/<name>/project.hcl
  3. Global local variables in ~/.dex/local.hcl
  4. Package defaults in package.hcl

Publishing Plugins

Plugin Structure

my-plugin/
├── package.hcl           # Plugin manifest (required)
├── content/              # Shared content files
│   └── code-review.md
├── skills/               # Skill content
│   └── testing.md
├── commands/             # Command content
│   └── deploy.md.tmpl
├── rules/                # Rule content
│   └── security.md
└── scripts/              # Helper scripts
    └── setup.sh

Cross-Platform Plugins

Share content files across platforms. Write once, deploy to all:

package {
  name      = "code-review-tools"
  version   = "1.0.0"
  platforms = ["claude-code", "github-copilot", "cursor"]
}

claude_skill "code-review" {
  description = "Thorough code review capability"
  content     = file("content/code-review.md")
}

copilot_skill "code-review" {
  description = "Thorough code review capability"
  content     = file("content/code-review.md")  # Same file!
}

Publishing Guide

Publish to Git

git tag v1.0.0
git push origin v1.0.0
plugin "my-plugin" {
  source  = "git+https://github.com/owner/my-plugin.git"
  version = "v1.0.0"
}

Best practices

  • Use descriptive, meaningful names for skills and commands
  • Document thoroughly. Include usage examples in content files.
  • Share content. Use file() to share content across platforms.
  • Use templates. Use templatefile() for platform-specific variations.
  • Pin dependencies. Use specific version ranges for stability.
  • Test on all platforms. Verify your plugin works on each target platform.
